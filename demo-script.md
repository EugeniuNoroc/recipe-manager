# Демо-сценарий защиты

Время: **7–10 минут**. Сервер запущен, БД и Redis в Docker.

---

## 0. Перед защитой (за 5 минут до)

```bash
docker ps                        # → оба контейнера Up: recipe-mysql, recipe-redis
```

- Открыть **http://localhost:8000** — убедиться, что сайт работает
- Разлогиниться, если был залогинен
- Открыть **второе окно терминала** с:

```bash
docker exec -it recipe-redis redis-cli
```

## 5. Redis-фичи (2 мин)

### 5.1 Счётчик просмотров

Перейти на `/view.php?id=1` — обновить страницу 3–4 раза.

В redis-cli:

```
GET recipe:1:views
```

→ Показывает число просмотров.

> «Каждый просмотр — атомарный INCR. Без блокировок, без записи в MySQL.»

### 5.2 Статистика

Перейти на `/stats.php` → виден топ рецептов по просмотрам.

> «Данные читаются через KEYS + MGET — один round-trip к Redis вместо N запросов к MySQL.»

### 5.3 Избранное

На странице рецепта нажать **♡ В избранное**.

В redis-cli:

```
SMEMBERS user:1:favorites
```

→ Показывает id рецепта в SET.

Перейти на `/favorites.php` — рецепт в списке.

> «Redis SET — O(1) для добавления/удаления/проверки. Не нужна отдельная таблица MySQL для избранного.»

## 6. Rate limiting (30 сек)

Создать 3 рецепта подряд быстро → четвёртая попытка:

> Flash: «Слишком много рецептов за короткое время. Подождите немного.»

В redis-cli:

```
GET rate:create:1
TTL rate:create:1
```

→ Счётчик = 4, TTL ~50с.

> «INCR + EXPIRE — атомарная операция, нет race condition. Лимит: 3 рецепта за 60 секунд на пользователя.»

---

## 7. Graceful Degradation ⭐ главный козырь (1 мин)

```bash
docker stop recipe-redis
```

Перейти на `/index.php` → **страница работает**, но:
- Бейдж **⚠ Redis offline** в навигации
- Баннер «Redis offline — избранное и статистика временно недоступны»
- Юзер «разлогинен» (сессия не читается — это ожидаемо)

Перейти на `/stats.php` → страница работает, топ пуст.
Перейти на `/view.php?id=1` → страница работает, просмотры = 0.

> «MySQL — источник правды, Redis — ускоритель. При падении Redis теряются только эфемерные данные.
> Ни одна страница не даёт 500-ошибку. Это обеспечивает класс SafeRedis — обёртка над Predis,
> которая перехватывает ConnectionException на каждом вызове и возвращает безопасные дефолты.»

```bash
docker start recipe-redis
```

Подождать 2–3 секунды, обновить страницу → бейдж исчез, можно логиниться снова.

> «Восстановление автоматическое: при следующем запросе bootstrap.php снова пингует Redis
> и, если он отвечает, создаёт SafeRedis с живым соединением.»

## 8. Вопросы — заготовки ответов

**Q: Почему Redis, а не Memcached?**
> Redis поддерживает разнообразные структуры данных (SET, ZSET, String), персистентность и TTL
> на уровне ключа. Для избранного (SET) и rate limiting это принципиально. Memcached — только String.

**Q: Что если Redis и MySQL не синхронизированы?**
> В нашей архитектуре MySQL — единственный источник правды для структурных данных.
> Redis хранит только то, что можно безопасно потерять: счётчики (пересчитаются),
> сессии (юзер перелогинится), избранное (в продакшне лучше дублировать в MySQL).

**Q: Predis vs phpredis?**
> phpredis — расширение C, быстрее. Predis — pure PHP, без компиляции, ставится через Composer.
> Для лабораторной работы Predis предпочтительнее: нулевая зависимость от конфигурации PHP.

**Q: Как работает SafeRedis?**
> SafeRedis оборачивает реальный Predis\Client. Каждый вызов (`__call`) завёрнут в try/catch
> для `ConnectionException` и `CommunicationException`. При первом сбое флаг `$alive` сбрасывается
> и все последующие вызовы идут в NullRedisClient (заглушку с нулевыми значениями).

**Q: CSRF — как работает?**
> При рендере формы генерируется случайный токен, сохраняется в Redis (TTL 1ч) или $_SESSION.
> Токен вставляется как скрытое поле `_token`. При POST-запросе `Csrf::verify()` сравнивает токены.
> `hash_equals` защищает от timing-атак.
