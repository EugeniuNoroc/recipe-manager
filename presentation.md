**Скриншот/визуал:** Диаграмму выше можно вставить как текст или перерисовать в Excalidraw / draw.io. Рекомендую сделать цветную версию: MySQL (синий), Redis (красный), PHP (фиолетовый).

**Что говорить (60 сек):**

> «Архитектура разделена на два уровня хранения. MySQL хранит структурированные данные:
> пользователей, рецепты, категории, теги — всё, что нельзя потерять.
> Redis хранит эфемерные данные: токены сессий с TTL 24 часа, CSRF-токены, счётчики просмотров,
> избранное как SET и rate-лимиты как счётчики с TTL.
>
> Почему две БД? MySQL гарантирует ACID и целостность через FK.
> Redis даёт O(1) для GET/SET и атомарный INCR — это принципиально для счётчиков под нагрузкой.
> Если Redis упадёт — приложение продолжит работать только из MySQL: пользователи увидят рецепты,
> но не смогут войти, а счётчики просмотров обнулятся. Это называется graceful degradation.»

## Слайд 6: Redis — ключи и структуры данных

**Что на слайде (текст/визуал):**

| Ключ | Тип | TTL | Назначение |
|---|---|---|---|
| `session:{token}` | String | 24 ч | Хранит user_id; заменяет файловые PHP-сессии |
| `csrf:{session_id}` | String | 1 ч | CSRF-токен; валидируется при каждом POST |
| `user:{id}:favorites` | **SET** | нет | Множество id рецептов в избранном |
| `recipe:{id}:views` | String | нет | Счётчик просмотров; `INCR` — атомарно |
| `rate:create:{user_id}` | String | 60 с | Rate limit; `INCR + EXPIRE` |

**Скриншот/визуал:** Скриншот из redis-cli с реальными ключами после взаимодействия с сайтом.

`[SCREENSHOT 6: redis-cli после логина и просмотра рецептов. Команды: KEYS * → TTL session:xxx → SMEMBERS user:1:favorites → GET recipe:1:views]`

**Что говорить (45 сек):**

> «В Redis я использую три разных типа данных.
> String для счётчиков и сессий: INCR атомарно увеличивает значение без race condition.
> SET для избранного: SADD, SREM, SISMEMBER — все O(1).
> TTL позволяет не чистить данные вручную: сессии истекают через 24 часа автоматически,
> rate-лимит сбрасывается через 60 секунд.
> Это то, что было бы сложно и медленно реализовать на чистом MySQL.»

## Слайд 7: 7 функций приложения

**Что на слайде (текст/визуал):**

1. **Авторизация** — регистрация / логин / логаут (bcrypt + Redis session + cookie)
2. **CRUD рецептов** — создание, просмотр, редактирование, удаление (только автор)
3. **Категории и теги** — many-to-many через recipe_tags; теги создаются на лету
4. **Поиск и фильтрация** — по названию/ингредиентам (LIKE) И категории (WHERE, AND)

6. **Статистика просмотров** — Redis INCR на каждый просмотр, топ-10 по просмотрам
7. **Rate limiting** — не более 3 рецептов за 60 секунд на пользователя

## Заготовки ответов на типичные вопросы

---

### Q: «Зачем вам Redis, если всё можно хранить в MySQL?»

**Ответ:**

Технически — да, всё можно реализовать в MySQL. Но Redis решает конкретные задачи лучше.
Во-первых, MySQL не предназначен для хранения сессий с автоматическим истечением: нужна отдельная таблица и cron для очистки. Redis делает это через TTL на уровне ключа — никакой дополнительной логики.
Во-вторых, атомарный `INCR` в Redis — это одна операция без транзакций и блокировок. Аналог в MySQL: `UPDATE counters SET views = views + 1 WHERE recipe_id = ?` — это блокировка строки при конкурентных запросах.
В-третьих, Redis SET для избранного позволяет O(1) для SADD, SREM, SISMEMBER. В MySQL пришлась бы отдельная таблица и JOIN.
Итог: я использую каждый инструмент для того, для чего он создан.

### Q: «Что произойдёт, если Redis недоступен?»

**Ответ:**

Приложение переходит в режим деградированной работы. Ни одна страница не вернёт 500-ошибку. Это обеспечивает класс `SafeRedis` — обёртка над `Predis\Client`, которая перехватывает `ConnectionException` и `CommunicationException` на каждом вызове и возвращает безопасные дефолты: `null`, `0`, `[]`. Параллельно используется `NullRedisClient` — заглушка, которая возвращает те же дефолты без попыток подключения.
Конкретные последствия: пользователи выходят из системы (сессии в Redis), избранное недоступно, счётчики просмотров обнуляются. Но список рецептов, поиск, фильтрация — всё, что читается из MySQL — продолжает работать. CSRF переключается на `$_SESSION` (нативные PHP-сессии). При восстановлении Redis приложение возвращается в полный режим автоматически на следующем запросе.

### Q: «Как синхронизируются данные между MySQL и Redis?»

**Ответ:**

В данной архитектуре синхронизация не нужна, потому что данные не дублируются. MySQL и Redis хранят разные сущности. MySQL — структурные данные: рецепты, пользователи, теги. Redis — эфемерные: сессии, счётчики, избранное. Счётчик просмотров в Redis — это отдельная информация, которая не пишется обратно в MySQL. Если бы нам нужна была персистентная статистика, пришлось бы либо писать в MySQL асинхронно через очередь, либо периодически синхронизировать через cron. В лабораторной работе такое требование не стоит, поэтому счётчики в Redis — это просто быстрый кэш, который можно безопасно потерять.

### Q: «Почему Predis, а не phpredis?»

**Ответ:**

phpredis — это C-расширение для PHP. Оно быстрее, но требует установки через `pecl install redis` и перекомпиляции PHP, что не всегда возможно в учебных или облачных средах. Predis — pure-PHP библиотека, ставится через `composer require predis/predis` без каких-либо дополнительных зависимостей. Для лабораторной работы это принципиально: установка воспроизводима на любой машине одной командой. По производительности разница ощутима под нагрузкой (тысячи запросов в секунду), но не в учебном приложении. Если бы это был production-проект с высокими требованиями к latency, выбор был бы в пользу phpredis.

### Q: «Почему Docker, а не нативная установка MySQL?»

**Ответ:**

Docker решает три проблемы сразу. Первая — воспроизводимость: `docker run mysql:8` на любой машине даёт одинаковую среду, без зависимости от версии ОС или конфигурации хоста. Вторая — изоляция: MySQL работает в контейнере и не конфликтует с другими версиями MySQL или другими сервисами. Третья — скорость: не нужно устанавливать, настраивать и сносить сервисы вручную. Для демо и защиты это особенно важно: `docker start recipe-mysql recipe-redis` — и среда готова за 5 секунд. `docker stop recipe-redis` — и можно наглядно показать graceful degradation.

### Q: «Как обеспечивается безопасность паролей?»

**Ответ:**

Пароли хранятся исключительно в виде bcrypt-хеша через стандартную функцию PHP `password_hash($password, PASSWORD_BCRYPT)`. bcrypt намеренно медленный: одна операция хеширования занимает ~100-300 мс, что делает перебор по словарю экономически нецелесообразным даже при утечке БД. Верификация — через `password_verify($input, $storedHash)`, который использует timing-safe сравнение. Сырой пароль нигде не логируется и не хранится в сессии. В Redis хранится только `user_id`, который читается из cookie-токена: даже если Redis скомпрометирован, пароли не утекают.

### Q: «Зачем CSRF-токен, если есть SameSite cookies?»

**Ответ:**

SameSite=Lax (который я использую) защищает от большинства CSRF-атак через cross-site запросы, но не от всех сценариев. Например, GET-запросы с cross-site навигации (переход по ссылке) с SameSite=Lax всё равно отправляют cookie. CSRF-токен добавляет второй уровень защиты, не зависящий от поведения браузера. В соответствии с принципом "defence in depth" (эшелонированная защита) лучше иметь два независимых механизма, чем один. Также CSRF-токен в Redis с TTL 1 час решает проблему устаревших форм: если пользователь не закрыл вкладку сутки, токен всё равно истечёт.

### Q: «Что такое Sorted Set в Redis и почему ZINCRBY — это O(log N)?»

**Ответ:**

Sorted Set (ZSET) — это структура данных Redis, в которой каждый элемент хранится вместе с числовым "score" (баллом). Элементы автоматически сортируются по score. `ZINCRBY key increment member` атомарно увеличивает score элемента. Сложность O(log N), потому что внутри ZSET реализован на skip list (список с пропусками) — структура, которая при вставке/обновлении требует обхода на нескольких уровнях за логарифмическое время, аналогично дереву поиска. N — количество элементов в ZSET. Для топ-10 из миллиона рецептов: `ZREVRANGE key 0 9` — O(log N + M), где M = 10. MySQL-аналог (`ORDER BY views DESC LIMIT 10`) потребовал бы либо полного scan, либо индекса, который нужно поддерживать при каждом UPDATE.

### Q: «Как работает rate limiting и можно ли его обойти?»

**Ответ:**

Rate limiting реализован на основе Redis `INCR` + `EXPIRE`. При каждом запросе на создание рецепта делается `INCR rate:create:{user_id}`. Если счётчик только что создан (значение = 1), устанавливается `EXPIRE 60`. Если значение превышает лимит (3), запрос отклоняется. `INCR` в Redis атомарен — нет race condition при конкурентных запросах. Обойти можно несколькими способами: создать новый аккаунт (лимит привязан к user_id), использовать несколько сессий, или, теоретически, атаковать в момент перехода между окнами TTL (sliding window exploit). Для production-системы применяют алгоритм "token bucket" или "sliding window log" с более сложной логикой. В данной реализации это fixed window counter — достаточный для учебной задачи.

### Q: «Почему сессии в Redis, а не в файлах?»

**Ответ:**

Стандартные PHP-сессии хранятся в файлах на диске (обычно в `/tmp`). У этого подхода несколько проблем. Во-первых, скорость: доступ к файловой системе медленнее, чем к Redis в памяти, особенно при высокой нагрузке. Во-вторых, масштабирование: если запустить несколько PHP-процессов на разных серверах, сессии в файлах не шарятся между ними — нужен sticky session или shared filesystem. Redis решает обе проблемы: сессия читается за < 1 мс и доступна из любого узла кластера. В-третьих, TTL: в Redis `SETEX session:{token} 86400 user_id` автоматически удалит ключ через 24 часа — нет необходимости в сборщике мусора. Дополнительно: кастомные токены в cookie более безопасны, чем стандартный `PHPSESSID`, который предсказуем по формату.
